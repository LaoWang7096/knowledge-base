Codex 必装十大 Skills，我挨个翻车之后，重新排了一次顺序

原创 Dr.Joyi Dr.Joyi 像素与咖啡时光 2026年5月27日 11:11 江苏

早上 10 点 15 分，我刚开完一个客户晨会，正打开电脑想顺一遍今天要交付的脚本。

朋友群弹了一张图。

Codex 必装十大 Skills，封面做得很精致， Superpowers 高居榜首，每个 Skill 后面挂着 GitHub 星标数。

第一反应跟你大概一样：又来。

Skill 清单这种东西，我这半年看吐了。 Claude Code 阵营每隔两周就出一份必装清单，每一份我都装过一遍，每一份最后留下来的不到一半。

但我点开仔细看了三分钟。

我突然意识到，这张图跟之前所有的清单，有一个本质区别。

这次的清单，是 Codex 的"专属文化"

你可能没意识到这件事：

Codex 一直没有 Skills 生态。

Claude Code 从去年开始疯狂长 skills 系统，社区一波一波堆。 Codex 这边呢，一直是赤手空拳——靠 prompt 、靠 system message 、靠 codex.json那点配置项硬扛。

我自己就吃过这个亏。

去年下半年，我给一个客户做数据清洗工具。整个项目结构不复杂，但 Codex 有个特别让人难受的毛病：它会擅自简化你的需求。

第一版我写完丢给它，跑了，崩；告诉它哪里错了，它改完跑，又崩；改到第五版，问题已经不是逻辑了，是它每次重写都偷偷把我定义的几个边界字段给删了。

我当时一度想砸键盘。

最后我把整个工作流挪去 Claude Code + Superpowers ，第一版直接跑通。

那时候我以为是 Claude Code 比 Codex 强。

后来我才反应过来：不是。

是 Superpowers 把 TDD 流程强制锁死了——它逼着 agent 先写测试，再写代码，再过审。这个护栏不是模型给的，是 Skill 给的。

而 Codex 当时根本接不上这层护栏。

——所以今天这张图最重要的信号，不是上面这 10 个 Skill 多牛逼。

是 Codex 终于开始有自己的 Skills 文化了。

下面这一周，我把这 10 个全装了一遍，挨个用，挨个翻车，重新排了一份我自己的优先级。

第一档：天天上场，已经长进 Codex 的护栏（ 2 个）

1. Superpowers——让 Codex 不再"擅自简化"的那根缰绳

这个排第一名，没意见。

但封面图给的理由是强制 TDD + 代码审查，太抽象。

我给你说人话：这玩意儿能让 Codex 在干活的过程中，被迫慢下来。

Codex 模型本身有个倾向，叫快速给你一个答案。这个倾向在简单任务上是优点，在复杂任务上是灾难。

Superpowers 在 Codex 上做的事，就是在 Codex 想跳到答案的那一秒，把它按回去——你得先列计划，得先写测试，得先 review 。

我现在写任何超过 100 行的 Codex 脚本，没装 Superpowers 我就不敢开工。

不是因为我离不开它，是因为离开它之后我得自己人肉当那个监工。我太懒了，做不到。

2. Planning with Files——Codex 上下文窗口的外挂硬盘

这是个被严重低估的 Skill 。

如果说 Claude Code 的上下文是别墅， Codex 的上下文就是个开间——窄。窄到你跑一个稍微大一点的项目，前面给它的需求它后面就忘了。

Planning with Files 干的事很朴素：把任务计划、 findings 、 progress 全部落地成 markdown 文件。

Codex 每跑一步，先读文件、再做事、做完更新文件。

听起来很笨。

但笨就是它的核武器。

我有一次让 Codex 做一个 7 步的数据流水线，全程没用 Planning with Files——跑到第 4 步它已经忘了第 1 步定的输出格式，回头一查，三个表的列名全部对不上。

后来同样的任务我接上 Planning with Files ，跑完之后 progress.md 那个文件，比我自己写的项目周报还清楚。

Codex 没有上下文耐力，但 markdown 文件有。

这个 Skill 让 Codex 第一次像个能干长线活的人。

第二档：特定场景下的神器（ 3 个）

3. SuperClaude Framework——30 条 slash 命令的弹药库

它不是一个 Skill ，是一套指挥语法。

/analyze、/refactor、/explain、/document——把你平时要打一大段 prompt 才能讲清楚的事，浓缩成一条斜杠命令。

实战感受：前两周很爽，第三周开始过载。

30 条命令我记不全。最后真正变成我肌肉记忆的，就 4 条：/analyze、/refactor、/test、/explain。剩下 26 条对我来说只是知道它在那儿。

但这 4 条，每天能给我省 15 分钟的 prompt 时间。

值这一装。

4. Anthropic Official Skills——参考实现的标准答案

这个 Skill 包里最有用的，不是它能干什么。

是它告诉你官方认为一个 Skill 应该长什么样。

里面有几个 skill 是真的参考级别的——比如 skill-creator，比如几个文档处理 skill 的结构。如果你以后要自己造 Skill （不管在 Codex 还是 Claude Code ），先读这个包里的源码，比读任何教程都管用。

我自己造的几个工作流 Skill ，模板全是从这里抄来的，我强烈建议自己创建skills的，一定要使用 skill-creator，真个是真牛逼。

5. Vercel Agent Skills——React 和前端的 140+ 条审查规则

这是个前端工程师的护身符。

Vercel 把他们自己内部 review React 代码的规则整套打包成 Skill ，让你的 Codex 在写前端的时候被这套规则审一遍。

我自己不是前端开发，所以用得不多。但我做内部工具的时候——比如给客户做一个数据可视化页面——挂上这个 Skill ， Codex 写出来的代码就不会犯那些工程师一眼能看出来的低级错。

排版乱、命名乱、 useEffect 滥用，这些事它能帮你提前堵掉。

第三档：装了，还在摸索（ 3 个）

6. MiniMax Skills——10 个工业级流程卡

老实说，我没深用。

它是个流程卡集合：前端、全栈、移动端、文档生成各一套。我装上之后跑了两次，输出确实工业感很强——目录结构整齐、配置文件完整、 README 都写好了。

但它对我这种做内容/客户咨询的来说，杀鸡用了牛刀。

如果你是真在做产品交付，比如要给客户搭一个完整的小程序，这个 Skill 应该是你的主力。

我不是这个语境，我说不准。

7. Context Engineering Skills——教 Codex 管理自己的上下文

这个 Skill 的卖点很哲学：让 Codex 学会自己整理记忆。

实战感受：功能存在感低，但效果有。

我装上之后没觉得 Codex 突然变聪明了。但跑长任务的时候——3 小时以上那种——我能明显感觉到它不那么容易跑偏了。它会自己摘要、自己回溯、自己丢掉旧上下文。

但这个改善是隐性的。我没法跟你保证它在你的场景下也能感觉到。

它适合的是已经在跑大型任务的人，不是新手。

8. Composio Skills——MCP 和 Skills 的双层桥

这个最尴尬。

它的定位很高级：让你的 Codex 通过 MCP 调用外部服务，再通过 Skills 把这些调用流程沉淀下来。

听起来很美。

但 MCP 这个生态本身在 Codex 上还不算成熟。我装上 Composio 之后，能调通的服务不到一半——剩下那一半要么文档对不上，要么权限对不上，要么版本对不上。

我对它的态度是：它代表了正确的方向，但现在还不是它的时间。

半年后回来再看。

第四档：当目录翻，不当工具用（ 2 个）

9. Antfu Skills——看看高手是怎么写 Skill 的

Antfu 是前端圈的传奇人物。

他的 Skills 包，与其说是装来用，不如说是装来读源码学手艺。

我装上之后没真的调用过几次。但每次我自己造新 Skill 卡壳了，我会去翻一下他的实现——尤其是他怎么处理 metadata 、怎么写 trigger 词、怎么组织文件结构。

学手艺的，比解决问题的好用。

10. Awesome Agent Skills——500+ Skill 的目录索引

这是个索引，不是工具。

它的作用是当你想找一个特定场景的 Skill 时——比如我想找一个专门做 SEO 优化的——你可以来这里翻。

我自己用得不多。因为我有 find-skills 这个 Skill 直接在本地翻。

但如果你刚入门、还没有 find-skills 、还在选型阶段， Awesome Agent Skills 是你最该收藏的浏览器书签之一。

我重新排出来的顺序，跟朋友圈相差挺远

封面把 Superpowers 放第一是对的。但第二到第十的排序，我不同意。

我自己用下来的真实优先级是这样：

先装 1+2： Superpowers 装护栏， Planning with Files 装外挂硬盘。这两个直接改变 Codex 的工作底盘。

再补 3+4+5： SuperClaude 给你命令语法， Anthropic Official 给你 Skill 模板， Vercel Agent 给你前端审查。这三个是按需调用，不是天天用。

剩下 5 个，按需观察： MiniMax 看你做不做产品交付； Context Engineering 看你跑不跑长任务； Composio 等半年再看； Antfu 当教科书； Awesome Agent 当书签。

如果你 Codex 用了不到三个月，只装 1+2 就够了。

剩下的别一次装满，那只会让你的 Codex 启动变慢、回答跑偏、互相冲突。

我亲眼见过有人装了 30 个 Skill ，结果 Codex 一启动光加载 Skills 就要 8 秒。每次让它干活，它都得先回忆一遍我都有哪些工具。

装得越多，每个 Skill 能分到的注意力越少。

这是个反直觉的事实，但越用越确定。

写在最后

Codex 这一年走得很拧巴。

它有顶尖的代码模型，但没有顶尖的工作流文化。它有疯狂的执行速度，但没有疯狂的审慎流程。

它一直是个赤手空拳的天才——能力强，但没护具。

今天这张十大 Skills 的清单，是它第一次开始穿装备。

Superpowers 是它的护甲， Planning with Files 是它的腰包， SuperClaude 是它的腰带武器槽，剩下的 Skill 各种小道具。

这套装备穿齐了， Codex 才真正能下场跟 Claude Code 五五开。

而在此之前，它一直只是一把锋利的剑，但握剑的人会受伤。

我做了一年 AI 工具的人来说一句不客气的话：

工具的进化，从来不是模型变强。是围绕模型的文化变厚。

Codex 终于有了自己的 Skills 文化。

至于装哪几个、按什么顺序装——别全信朋友圈推荐，别全信我。

去试。试一周，再决定哪个留下来。
